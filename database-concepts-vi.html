A

Database là gì? Database

Một cơ sở dữ liệu (database) là tập hợp dữ liệu có liên quan với nhau, được lưu trữ có tổ chức và có thể truy xuất hiệu quả.

  • Dữ liệu văn bản & số: tên, địa chỉ, điểm số, giá sản phẩm…
  • Đa phương tiện: hình ảnh, âm thanh, video (multimedia database)
  • Thông tin địa lý: bản đồ, tọa độ, vùng lãnh thổ (GIS database)
💡 Mẹo nhớ

Hãy hình dung database như một tủ hồ sơ siêu thông minh — không chỉ lưu giấy tờ mà còn biết tìm kiếm, sắp xếp và bảo vệ dữ liệu tự động.

B

Thuật ngữ cơ bản Key Terms

Data
Các sự kiện thô chưa được xử lý. VD: "Nguyễn Văn A", 85 điểm.
Meta-data
Dữ liệu mô tả dữ liệu: tên cột, kiểu dữ liệu, ràng buộc...
DBMS
Phần mềm quản lý CSDL (MySQL, Oracle, SQL Server, PostgreSQL).
Database System
= Database + DBMS + Ứng dụng + Người dùng.
Application Program
Phần mềm giao tiếp với DBMS để thao tác dữ liệu cho người dùng cuối.
🧠 Câu hỏi hay gặp
  • DBMS ≠ Database: DBMS là phần mềm, Database là kho dữ liệu. MySQL là DBMS; còn "bảng sinh viên" là database.
  • Meta-data lưu trong "data dictionary" hay "catalog" của DBMS.

Đặc điểm của Database Characteristics

Đặc điểmGiải thíchVí dụ
Program-Data Insulation
(Tách biệt chương trình & dữ liệu)		 Thay đổi cấu trúc lưu trữ không ảnh hưởng tới chương trình ứng dụng. Thêm cột "email" vào bảng mà không cần sửa code cũ.
Data Abstraction
(Trừu tượng hóa dữ liệu)		 Người dùng thấy giao diện đơn giản, ẩn đi chi tiết lưu trữ phức tạp bên trong. Bạn SELECT tên sinh viên mà không cần biết file lưu trên ổ đĩa nào.
Multiple Views Mỗi nhóm người dùng có thể thấy "góc nhìn" khác nhau của cùng một dữ liệu. Giảng viên thấy điểm toàn lớp; sinh viên chỉ thấy điểm của mình.
Data Sharing & Multiuser Nhiều người cùng truy cập đồng thời mà không xung đột dữ liệu. Đặt vé máy bay: hàng trăm người tra cứu cùng lúc.
💡 Mẹo nhớ

Insulation = áo giáp — chương trình được "bọc" khỏi thay đổi lưu trữ.
Abstraction = kính râm — nhìn đơn giản hơn thực tế phía sau.

D

Người dùng Database Database Users

  • Database Administrator (DBA): Quản trị toàn bộ hệ thống — phân quyền, backup, tối ưu hiệu năng, cài đặt DBMS.
  • Database Designer: Thiết kế cấu trúc CSDL (bảng, quan hệ, ràng buộc) dựa trên yêu cầu nghiệp vụ.
  • Application Programmer: Viết code để tương tác với DBMS (Java, Python, PHP…).
  • End Users (Naive): Người dùng cuối không cần biết SQL, chỉ dùng giao diện ứng dụng.
  • End Users (Sophisticated): Người dùng thành thạo, có thể tự viết truy vấn SQL.
  • System Analyst: Phân tích yêu cầu hệ thống, cầu nối giữa người dùng và lập trình viên.
🎯 Mẹo ghi nhớ thứ tự vai trò

Nhớ theo luồng dự án: Analyst → Designer → Programmer → DBA → End User. Từ phân tích yêu cầu → thiết kế → xây dựng → vận hành → sử dụng.

Data Model Data Model

Data Model = Cấu trúc (Structure) + Ràng buộc (Constraints) + Thao tác (Operations)

Cấu trúc dữ liệu Ràng buộc toàn vẹn Thao tác CRUD
LoạiMô tảVí dụ
Conceptual Mô hình gần với tư duy người dùng, ẩn chi tiết vật lý. Dùng để giao tiếp với khách hàng. ER Model, EER Model
Implementation
(Representational)		 Cân bằng giữa khái niệm và vật lý. DBMS thực tế sử dụng. Relational Model (bảng), Network, Hierarchical
Physical Mô tả cách dữ liệu thực sự lưu trên ổ đĩa (file, index, block). B-tree index, heap file
💡 Mẹo nhớ

Nhớ bằng câu: "Con Iguana Phải chạy" — Conceptual → Implementation → Physical, mức độ trừu tượng giảm dần.

F

Database Schema & State Schema vs. State

Database Schema
Bản mô tả cấu trúc CSDL — định nghĩa bảng, cột, kiểu dữ liệu. Ít thay đổi.
Database State
Dữ liệu thực tế trong CSDL tại một thời điểm. Thay đổi liên tục.
Schema Diagram
Biểu đồ trực quan hóa schema (thường là sơ đồ bảng và quan hệ).
Instance
Một snapshot của database state tại một thời điểm cụ thể.
🧠 Ví von dễ nhớ

Schema = Bản vẽ ngôi nhà (blueprint) — cố định, không đổi khi người vào ra.
State = Tình trạng ngôi nhà hôm nay — có thể có người, không có người, đồ đạc thay đổi.

G

Kiến trúc 3 lớp Schema Three-Schema Architecture

Mục tiêu: tách biệt ứng dụng người dùng khỏi lưu trữ vật lý, đảm bảo tính độc lập dữ liệu.

External
View của từng nhóm người dùng. Mỗi người thấy một "góc nhìn" khác nhau.
⟷ Logical Data Independence
Conceptual
Toàn bộ cấu trúc logic của CSDL — bảng, ràng buộc, quan hệ. Ẩn chi tiết lưu trữ.
⟷ Physical Data Independence
Internal
Cách dữ liệu thực sự lưu: file, index, block trên ổ đĩa vật lý.
Loại độc lậpNghĩa là
Logical Data Independence Thay đổi Conceptual Schema không ảnh hưởng đến External Views và ứng dụng.
Physical Data Independence Thay đổi Internal Schema (cách lưu trữ) không ảnh hưởng đến Conceptual Schema.
💡 Mẹo nhớ vị trí

Nhớ từ dưới lên: Internal (ổ đĩa) → Conceptual (logic) → External (người dùng). Viết tắt: ICE — lạnh từ dưới lên ❄️

H

Ngôn ngữ DBMS DBMS Languages

Ngôn ngữTên đầy đủChức năngLệnh tiêu biểu
DDL Data Definition Language Định nghĩa & chỉnh sửa cấu trúc CSDL (schema) CREATE, ALTER, DROP, TRUNCATE
DML Data Manipulation Language Thao tác dữ liệu: thêm, sửa, xóa, truy vấn SELECT, INSERT, UPDATE, DELETE
DCL Data Control Language Kiểm soát quyền truy cập GRANT, REVOKE
TCL Transaction Control Language Quản lý giao dịch COMMIT, ROLLBACK, SAVEPOINT
🎯 Phân biệt nhanh DDL vs DML
  • DDL = Xây nhà (tạo bảng, xóa bảng). Tác động đến cấu trúc.
  • DML = Sống trong nhà (thêm/sửa/xóa dữ liệu). Tác động đến nội dung.

Kiến trúc DBMS Centralized & Client–Server

Centralized DBMS: Toàn bộ xử lý và lưu trữ trên một máy chủ duy nhất. Đơn giản nhưng không mở rộng được.

Kiến trúcĐặc điểmỨng dụng thực tế
Centralized Mọi thứ trên 1 máy. Đơn giản, dễ quản lý nhưng không scale được. Hệ thống nhỏ, nội bộ
2-Tier Client–Server Client giao tiếp trực tiếp với DB Server qua ODBC/JDBC. Ứng dụng desktop doanh nghiệp nhỏ
3-Tier Client–Server Client → App Server → DB Server. Bảo mật tốt hơn, dễ mở rộng. Web app, e-commerce, ngân hàng
Distributed DBMS Dữ liệu phân tán trên nhiều máy/địa điểm, vẫn như một CSDL thống nhất. Hệ thống toàn cầu (Google, Amazon)
💡 Mẹo nhớ

3-Tier = Màn hình – Não – Kho: Client (màn hình hiển thị) → App Server (não xử lý) → DB Server (kho lưu trữ).
